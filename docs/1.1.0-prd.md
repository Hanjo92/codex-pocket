# 1.1.0 PRD — attention and triage

## Summary

`1.1.0` is the first post-public release focused on making `codex-pocket` useful as a **Codex triage dashboard** instead of just a lightweight remote viewer.

If the official Codex app is strongest at direct remote control, `codex-pocket` should become strongest at:

- scanning many threads quickly
- spotting which threads need human attention
- reducing noise before the user opens a full transcript

## Problem

The current UI can list threads and open a transcript, but it still treats most threads as roughly equal.

That creates a few gaps:

- users cannot quickly tell which threads are still running vs blocked vs done
- important threads are mixed with low-priority recent activity
- project-level oversight exists, but actionable triage is still weak

## Goal

Make the home experience answer this question quickly:

> "What needs my attention right now?"

## Non-goals

Not for `1.1.0`:

- semantic transcript summarization (`1.2.0`)
- role/capability scoping (`1.3.0`)
- native app packaging
- public-internet deployment hardening beyond current baseline

## User outcomes

After `1.1.0`, a user should be able to:

1. open `codex-pocket`
2. immediately distinguish running, blocked, failed, and completed threads
3. switch to an inbox/attention view
4. see the most actionable threads first
5. jump into the right thread with fewer taps

## Release scope

### Included

#### #1 Thread/session state classification
- normalized state model for each visible thread
- safe fallback for ambiguous cases
- state surfaced in API and live refresh flow
- visible state badges in thread list and selected-thread UI

#### #2 Attention inbox
- dedicated view for attention-needed threads
- prioritization of blocked or actionable threads
- project-level counts/rollups aligned with normalized state
- mobile-friendly presentation

### Deferred

- per-thread natural-language summaries
- "what changed since last check" summary text
- permission scopes

## State model for `1.1.0`

Target states:

- `running`
- `waiting_input`
- `waiting_approval`
- `completed`
- `failed`
- internal fallback: `unknown`

`unknown` is useful as an implementation safety valve even if the main UI does not emphasize it.

## Attention model

Initial triage priority recommendation:

1. `failed`
2. `waiting_approval`
3. `waiting_input`
4. `running`
5. `completed`
6. `unknown`

Recommended default inbox behavior:

- show `failed`, `waiting_approval`, and `waiting_input` by default
- optionally allow `running` to appear in a secondary section
- keep `completed` in normal browsing, not primary inbox focus

## UX changes

### Thread list
- add compact state badge per thread
- preserve existing project-first navigation
- allow users to scan without opening each thread

### Sticky context / selected thread header
- show current normalized state
- keep state visible during live updates

### Inbox view
- add a toggle between:
  - all threads
  - needs attention
- sort actionable threads by severity and recency

### Project rollups
- expose counts like:
  - attention needed
  - running
  - completed recently

## API/runtime changes

### Thread payloads
Add a normalized state field to public thread/session payloads.

Candidate shape:

```json
{
  "id": "thread_123",
  "title": "Fix login bug",
  "projectLabel": "my-app",
  "state": {
    "type": "waiting_input",
    "label": "waiting_input",
    "updatedAtMs": 1715700000000
  }
}
```

Minimal alternative:

```json
{
  "state": "waiting_input"
}
```

Recommendation: start with the minimal string form unless the UI immediately needs more metadata.

### Live updates
Normalized state should refresh through the existing SSE/session update path without requiring full page reload.

## Success criteria

`1.1.0` is successful if:

- every visible thread gets a stable normalized state or safe fallback
- the thread list clearly shows state
- users can open an inbox/attention view that meaningfully reduces noise
- state changes appear during normal refresh/live update flows
- mobile usability is preserved

## Recommended implementation order

1. **#6** Define normalized state derivation rules
2. **#7** Expose normalized state in APIs and live updates
3. **#8** Show normalized state badges in UI
4. **#2** Build the attention inbox on top of the normalized state model

## Risks

### 1. Ambiguous raw signals
Codex state may not always cleanly reveal whether a thread is blocked, failed, or merely idle.

Mitigation:
- prefer explicit conservative rules
- use `unknown` instead of overclaiming

### 2. UI overcomplication
Too many colors/badges could make the list noisier instead of clearer.

Mitigation:
- keep badges compact
- reserve strongest emphasis for actionable states

### 3. Poll/live update mismatch
Thread list state may drift if only transcript/session details refresh.

Mitigation:
- ensure thread list refresh cadence and session-event update flow stay compatible
- consider lightweight list invalidation on selected-thread changes if needed

## Notes for later releases

- `1.2.0` can build natural-language summaries on top of the same state model
- `1.3.0` should use the same action classification when deciding which controls are allowed per role
